# 后端开发规范

## 1. 概述

本规范基于当前后端项目的代码风格和主流企业级项目开发实践，旨在确保团队开发的代码质量一致、可维护性高、安全性强。所有开发人员（包括AI辅助开发）在进行后端功能开发时必须遵守本规范。

## 2. 技术栈

### 2.1 核心技术
- Spring Boot 4.0.2+
- Java 21+
- MyBatis + PageHelper
- MySQL 9.5.0+
- Redis (缓存)
- RabbitMQ (消息队列)
- Spring Security + OAuth2 (认证授权)
- JWT (令牌管理)
- MinIO (对象存储)
- WebSocket (实时通信)

### 2.2 工具库
- Lombok
- Hibernate Validator
- Jackson
- Apache Commons Pool2

## 4. 命名规范

### 4.1 包名
- 采用小写字母，单词之间用点分隔
- 遵循Java包命名规范：`com.inkstage.模块名`

### 4.2 类名
- 采用大驼峰命名法
- 类名应清晰表达其职责
- 控制器：`XxxController`
- 服务接口：`XxxService`
- 服务实现：`XxxServiceImpl`
- 实体类：`Xxx`（与数据库表对应）
- DTO：`XxxDTO`
- VO：`XxxVO`
- 异常类：`XxxException`
- 工具类：`XxxUtils`

### 4.3 方法名
- 采用小驼峰命名法
- 方法名应清晰表达其功能
- 动词开头：如`createArticle`、`getUserInfo`
- 布尔返回值方法：`isXxx`、`hasXxx`

### 4.4 变量名
- 采用小驼峰命名法
- 变量名应具有描述性
- 避免使用单个字母（循环变量除外）
- 常量：全大写，下划线分隔

### 4.5 注释规范
- 类注释：使用`/** */`，说明类的职责
- 方法注释：使用`/** */`，说明方法功能、参数、返回值
- 代码注释：使用`//`，解释复杂逻辑
- 注释应清晰明了，避免冗余

## 5. 编码规范

### 5.1 代码风格
- 使用4个空格进行缩进
- 每行代码长度不超过120个字符
- 方法体长度控制在50行以内，超过则考虑拆分
- 类长度控制在500行以内，超过则考虑重构

### 5.2 依赖注入
- 使用构造函数注入，避免字段注入
- 配合Lombok的`@RequiredArgsConstructor`注解使用

### 5.3 异常处理
- 使用自定义异常类`BusinessException`处理业务异常
- 统一的全局异常处理器`GlobalExceptionHandler`
- 异常信息使用`ResponseMessage`中定义的常量

### 5.4 日志处理
- 使用SLF4J日志框架
- 类上添加`@Slf4j`注解
- 日志级别使用：
  - `debug`：调试信息
  - `info`：普通信息
  - `warn`：警告信息
  - `error`：错误信息
- 日志内容应包含关键参数信息

### 5.5 事务管理
- 使用`@Transactional`注解标记事务方法
- 明确指定`rollbackFor`属性
- 事务方法应尽量短小，避免长时间占用事务

### 5.6 安全规范
- 使用`@UserAccess`和`@AdminAccess`注解控制访问权限
- 敏感操作必须进行权限校验
- 密码等敏感信息必须加密存储
- 输入参数必须进行验证

### 5.7 参数规范
- **禁止使用Map作为方法参数和返回值**
- **禁止使用MapStruct等对象映射工具**
- **禁止使用魔法值**，所有常量必须在`constant`包中定义

### 5.8 类型声明规范
- **禁止使用`var`进行类型推断**，必须明确指定变量类型
- 例如：`UserEntity user = new UserEntity();` 而非 `var user = new UserEntity();`

### 5.9 变量命名规范
- **变量名应与类型名保持高度一致**，优先使用类型名或类型名的简化形式
- 当类型发生变化时，变量名必须同步更新，确保语义一致
- **反例**：
  ```java
  // 原 
  Long authorId = article.getAuthorId();
  // 新 
  Long authorId = article.getUserId();  // 错误：使用了authorId
  Integer count = article.getUserId(); // 错误：类型改变但变量名未同步
  ```
- **正例**：
  ```java
  Long userId = article.getUserId();  // 正确：与getUserId返回类型一致
  Long userId = user.getId();                  // 正确：与getId返回类型一致
  ```
- 避免使用无意义或与类型不匹配的变量名，如`temp`、`data`、`obj`等

### 5.10 代码质量原则
- **保持方法简洁**：单个方法行数控制在150行以内，职责单一明确
- **保证高可用性**：充分考虑边界条件、空指针、并发安全等因素，做好异常处理
- **保证高可读性**：代码逻辑清晰，命名规范，结构合理
- **禁止"屎山代码"**：
  - 禁止重复代码片段（DRY原则）
  - 禁止过长的方法链和嵌套层级
  - 禁止硬编码业务逻辑
  - 禁止未经测试的复杂逻辑
- 在编写代码时，应优先考虑可维护性和可扩展性，而非快速实现

### 5.11 分页处理
- **分页参数使用`PageRequest`**，可以选择继承或实现
- **分页响应使用`PageResult`**，可以选择继承或实现
- 分页参数必须包含`pageNum`和`pageSize`

## 6. 分层架构规范

### 6.1 控制器层（Controller）
- 负责接收和响应HTTP请求
- 进行参数验证和异常处理
- 调用服务层方法，返回统一的响应格式
- 不包含业务逻辑

### 6.2 服务层（Service）
- 负责业务逻辑实现
- 可以调用其他服务
- 处理事务和异常
- 向控制器层提供统一的业务接口

### 6.3 数据访问层（Mapper）
- 负责数据库操作
- 使用MyBatis XML或注解方式
- 不包含业务逻辑
- 禁止使用MyBatis Plus等ORM框架

### 6.4 数据传输对象（DTO）
- 用于控制器层和服务层之间的数据传输
- 包含请求参数和响应数据
- 使用`@Valid`注解进行参数验证
- **请求参数超过5个（不包含分页参数）必须使用DTO**
- **DTO字段必须与实体类字段一致**

### 6.5 视图对象（VO）
- 用于控制器层向客户端返回的数据结构
- 不包含敏感信息
- 结构清晰，便于前端使用
- **返回参数如果实体类不支持，必须使用VO**
- **VO字段必须与实体类字段一致**

## 7. 缓存规范

### 7.1 缓存使用
- 使用Redis作为缓存
- 缓存键使用`RedisKeyConstants`中定义的常量
- 缓存操作通过`RedisCacheManager`进行
- 数据变更时及时清除相关缓存

### 7.2 缓存策略
- 热点数据使用缓存
- 缓存过期时间合理设置
- 避免缓存穿透和缓存雪崩

## 8. 消息队列规范

### 8.1 消息生产
- 使用`NotificationProducer`发送消息
- 消息内容清晰明确
- 处理消息发送失败的情况

### 8.2 消息消费
- 使用`@RabbitListener`注解监听消息
- 实现幂等性处理
- 处理消息消费失败的情况


## 9. 文档规范

### 9.1 代码文档
- 类和方法必须有注释
- 复杂逻辑必须有解释
- 接口文档使用Swagger生成

### 9.2 架构文档
- 系统架构图
- 模块关系图
- 数据库设计文档

### 9.3 接口文档
- 接口说明
- 请求参数
- 响应格式
- 错误码说明


## 10. AI开发规范

### 10.1 代码生成
- 遵循本规范的所有要求
- 生成的代码应与现有代码风格一致
- 确保代码的可维护性和可读性

### 10.2 功能实现
- 理解业务需求，确保功能正确实现
- 考虑边界情况和异常处理

### 10.3 代码审查
- 生成的代码必须经过人工审查
- 确保代码符合规范和质量要求

## 14. 性能优化

### 14.1 数据库优化
- 使用索引
- 避免全表扫描
- 合理使用分页

### 14.2 代码优化
- 避免重复代码
- 优化算法和数据结构
- 减少不必要的计算

### 14.3 系统优化
- 使用连接池
- 合理设置线程池
- 优化缓存策略

## 15. 安全措施

### 15.1 认证授权
- 使用OAuth2和JWT
- 定期更新密钥
- 实现令牌过期机制

### 15.2 输入验证
- 对所有输入进行验证
- 防止SQL注入、XSS等攻击
- 限制请求频率

### 15.3 数据保护
- 敏感数据加密存储
- 传输数据使用HTTPS
- 定期备份数据

## 16. 总结

本规范旨在确保后端项目的代码质量、可维护性和安全性。所有开发人员（包括AI辅助开发）必须严格遵守本规范，以保证项目的长期稳定发展。

规范将根据项目的发展和技术的进步不断更新和完善。
